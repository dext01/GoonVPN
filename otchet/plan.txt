=== ЭТАП 0 — ПЕРВОНАЧАЛЬНЫЙ ПЛАН (ОТКЛОНЁН) ===

Февраль 2026. Исходное название проекта: WireTunnel.

Планировалось:
  - Протокол: WireGuard
  - Сервер: Ubuntu + WireGuard + iptables, подсеть 10.0.0.0/24
  - Клиент: Android (Kotlin + Jetpack Compose + wireguard-android)
  - Демо: подключение к локальному HTTP-серверу через туннель

Почему отказались:
  1. Хостинг mhost заблокировал VPN-трафик на уровне инфраструктуры —
     пришлось вернуть деньги и сменить провайдера.
  2. WireGuard блокируется российскими провайдерами через DPI —
     при поднятии туннеля SSH-соединение с сервером обрывалось
     и не восстанавливалось.


=== ЭТАП 1 — СМЕНА СТЕКА И НОВЫЙ ПЛАН ===

Решение:
  - Протокол заменён на VLESS + Reality (Xray-core).
    Reality маскирует трафик под обычный HTTPS — DPI не может
    отличить от легитимного соединения.
  - Клиентская библиотека заменена с wireguard-android на libXray.
  - Арендован новый VPS: Aeza, Стокгольм, Швеция (IP: 62.60.251.120).
  - Установлена панель 3x-ui для управления Xray-core через веб.
  - Создано подключение VLESS+Reality на порту 443.
  - Проверено через сторонний клиент Hiddify — работает.

Новая архитектура:
  Телефон → TUN-интерфейс (виртуальная сетевая карта)
           → HEV-socks5-tunnel (перехватывает IP-пакеты)
           → SOCKS5 прокси на 127.0.0.1:10808
           → Xray-core (шифрует в VLESS/Reality)
           → Сервер 62.60.251.120 (Швеция)
           → Интернет

=== ЭТАП 3 — ИСПРАВЛЕНИЕ БАГОВ ===

Всего найдено и исправлено 8 критических багов:

  БАГ 1 — VPN застрявал на «Подключение...»
    Причина: Android 14 блокирует broadcast без setPackage()
    Исправление: добавлен setPackage(packageName) в broadcastState()

  БАГ 2 — Режим маршрутизации сбрасывался при навигации
    Причина: хранился в Compose-состоянии, а не в SharedPreferences
    Исправление: чтение/запись через SettingsRepository

  БАГ 3 — Настройки blockBindToTun и allowLan не применялись
    Причина: XrayConfig.build() не принимал эти параметры
    Исправление: добавлены параметры allowLan и routingMode в build()

  БАГ 4 — Краш при отказе пользователя от VPN-разрешения
    Причина: RESULT_CANCELED не обрабатывался
    Исправление: onPermissionDenied() + Snackbar с инструкцией

  БАГ 5 — Журнал логов был пустым
    Причина: stdout Xray-процесса не писался в файл
    Исправление: добавлен поток записи в xray.log

  БАГ 6 — JNI-краш после попытки переименования пакета
    Причина: нативная библиотека скомпилирована под com.v2raytun...
    Решение: пакет оставлен как есть (невозможно переименовать
              без перекомпиляции нативной библиотеки)

  БАГ 7 — Маршрутизация не работала (только иллюзия)
    Причина: routing в Xray-конфиге был пустым
    Исправление: реальные правила domain/ip routing в XrayConfig

  БАГ 8 — Иконка приложения не менялась
    Причина: Android 8+ использует adaptive icon XML, а не webp
    Исправление: логотип помещён в drawable, XML обновлён

=== ЭТАП 7 — ПОДГОТОВКА К ЗАЩИТЕ ===

Что нужно знать наизусть:

  1. Стек технологий:
     TUN → HEV-socks5-tunnel → Xray-core → VLESS/Reality → сервер

  2. Что такое VLESS?
     Лёгкий протокол передачи трафика без лишней криптографии.

  3. Что такое Reality?
     Замена TLS — маскирует трафик под HTTPS к реальному сайту (SNI).
     DPI не отличает от легитимного трафика.

  4. Зачем HEV-socks5-tunnel?
     Android VpnService даёт только TUN file descriptor.
     HEV читает IP-пакеты из TUN и пробрасывает в SOCKS5-прокси Xray.

  5. Почему не переименовали com.v2raytun?
     JNI — нативная .so библиотека скомпилирована с именами функций
     по полному имени Java-класса. Переименование → UnsatisfiedLinkError.

  6. Почему не WireGuard?
     Блокируется российскими провайдерами через DPI.
     Reality невозможно заблокировать без ложных срабатываний.

  7. Как бот создаёт ключи?
     Через HTTP API 3x-ui → add_client → берёт uuid и срок → строит
     VLESS URL с публичным ключом Reality → отправляет пользователю.

Доказательная база (скриншоты):
  - Статус «Подключено» + таймер
  - 2ip.ru показывает IP Швеции
  - dnsleaktest.com без утечек
  - Разные IP в двух браузерах (split tunneling)
  - Экран логов с реальным выводом Xray
  - Telegram-бот: покупка → ключ → подключение
